# [상] 11차시 체험 효과 기록표 및 완성도 점검표

## 1. 체험 효과 기록표

본 시뮬레이터(이동약자 3D 시뮬레이터)를 직접 조작하며 느낀 장애물별 보행 체험 효과와 심리적 영향에 대한 기록입니다.

| 장애물 종류 | 체험 시 겪는 물리적 한계 (감각 피드백) | 실제 이동약자가 느낄 심리적 부담 및 스트레스 |
| :--- | :--- | :--- |
| **고중량 수동 통유리문** | - 키보드 스페이스바를 연타해야만 미세하게 열리며, 연타 시마다 스태미나가 감소함.<br>- 문을 완전히 밀기 전에는 통과할 수 없고, 속도가 절반(0.4m/s)으로 묶임. | - 뒤에 사람이 대기하고 있거나 짐을 들고 있을 때 극심한 불안감과 조급함을 유발함.<br>- 신체적 능력 한계가 타인에게 드러날 때 느끼는 사회적 위축감 및 좌절감. |
| **좁은 서가 구간** | - 마우스 시점을 조금만 삐끗해도 책장에 박혀 이동이 일시 정지되거나 멈춤.<br>- 휠체어의 회전 반경과 맞물려 미세 조작 스트레스가 발생함. | - 폐쇄된 공간에서 통행을 방해할까 봐 눈치가 보이고, 막다른 길이나 돌발 장애물을 만났을 때의 갇힌 듯한 패닉 현상. |
| **혼잡한 시민 동선 (NPC)** | - 불규칙하게 왕복하는 시민(NPC)들과 부딪힐 때마다 속도가 최저(0.42m/s)로 깎이고 관절 내구도가 지속 소모됨. | - 보행 흐름이 빠른 군중 속에서 언제 밀쳐질지 모르는 신체 위협과 보행 경쟁에서의 심각한 소외감 및 피로감. |
| **높은 의자 (바 체어)** | - 근처에서 기웃거리는 것만으로도 스태미나가 유의미하게 깎이며, 이동약자 전용 휴식처의 부재를 나타냄. | - 서점에 오래 머물고 싶어도 신체를 의지해 쉴 곳이 마땅치 않아, 적극적인 외부 활동 의지가 꺾이고 귀가를 조기에 결정하게 만듦. |
| **핸드레일 없는 계단** | - 계단 영역에 들어서자마자 화면이 핏빛으로 붉어지고 시야가 크게 흔들리며 스태미나가 초당 12.5%씩 소멸함. | - 손잡이 하나만 없어도 죽음이나 중상에 이를 수 있다는 극심한 공포감 및 물리적인 절벽을 만난 것 같은 막막함. |

### 비장애인 관점 vs 이동약자 관점 비교

*   **비장애인 기획자 관점**: "문이 조금 무겁고 서가가 좁긴 하지만 요령껏 천천히 비켜가면 계단까지 갈 수 있을 것이다."
*   **이동약자 체험자 관점**: "단 5초 동안 문을 열고 NPC 한 명과 살짝 부딪혔을 뿐인데 관절 체력이 완전히 바닥나 목적지(계단) 앞에서 발을 내딛을 수조차 없다. 계단에 손잡이가 없으니 눈앞에 두고도 갈 엄두가 나지 않는다."
*   **시뮬레이션을 통해 깨달은 점**: 이동약자에게 '어려운 보행 환경'은 단순한 불편함의 누적이 아니라 **'이동권의 원천적 박탈'과 '생명 위협'**으로 작용한다는 사실을 3D 감각 피드백(감속, 스태미나 고갈, 붉은 비네트)을 통해 뼈저리게 실감했습니다.

## 2. 완성도 점검표

시뮬레이터의 기술적 완성도와 기획 시나리오의 충실한 재현 여부를 평가한 점검표입니다.

| 평가 기준 | 자가 진단 및 평가 결과 | 개선 의견 및 해결 방안 |
| :--- | :--- | :--- |
| **1. 기획 시나리오와의 일치성** | **우수 (A)**<br>수동문 $\rightarrow$ 서가 $\rightarrow$ 시민 $\rightarrow$ 계단으로 이어지는 4단계 보행 장애물 체험 시나리오가 완벽하게 동선으로 녹아들어 있음. | - 현재 단일 경로인 가이드 라인을 확장하여, 장애물이 없는 배리어 프리(Barrier-Free) 대안 경로를 만들어 비교 체험하게 하면 효과가 더 극대화될 것 같음. |
| **2. 감각 피드백의 실감성** | **우수 (A)**<br>단순한 수치 표현을 넘어, 적색 비네팅, 카메라 셰이크, 지팡이 짚기 모션(`body.walking .cane` 키프레임 애니메이션) 등으로 신체적 한계를 눈과 손으로 느낄 수 있음. | - 관절염 환자가 겪는 통증이나 시야 흐림 현상을 묘사하기 위해 화면 색수차(Chromatic Aberration)나 오디오 노이즈 효과를 믹싱하면 몰입감이 향상될 것으로 보임. |
| **3. 물리/웹 기술의 안정성** | **보통 (B)**<br>Three.js WebGL 렌더러와 Cannon.js 물리 엔진이 매끄럽게 구동되며 충돌 처리(`collides`)가 오작동 없이 작동하나, 물리 조인트의 특성상 급조작 시 프레임이 튀는 경향이 있음. | - 물리 강체의 속성을 최적화하고 복잡한 HingeConstraint 회전 연산을 기하학적 수식 보간법으로 대체하여 저사양 디바이스에서도 60fps가 칼같이 유지되도록 코드 개선 필요. |
| **4. 시스템 인터페이스 가독성** | **우수 (A)**<br>화면 우측 하단의 실시간 미션 가이드와 좌측 상단의 관절 내구도(스태미나 게이지) 바가 유기적으로 작동하여 사용자가 목표를 바로 인지함. | - 모바일 패드 터치 조작 시 조작계 레이아웃이 화면 일부를 가리므로, 모바일 기기 감지 시 UI 스케일링을 자동 조절하도록 CSS 고도화. |
